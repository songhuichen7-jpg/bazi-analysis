## 一、排盘引擎选型

### 1.1 候选方案对比

| 库名 | 语言 | Stars | 核心功能 | 大小 | 许可证 |
|------|------|-------|---------|------|--------|
| **Lunisolar.js** | TypeScript | - | 八字四柱、五行纳音、200+ 神煞、节气、胎神 | GZIP 11.5KB | - |
| **mystilight-8char** | 纯 JS（零依赖） | - | 完整神煞系统、大运流年、十神、五行力量 | 轻量 | - |
| **bazi-mcp** | TypeScript | 339 | MCP 协议八字计算、真太阳时、早晚子时 | - | - |
| **shunshi-bazi-core** | TypeScript | - | 真太阳时校正、十神、五行、大运 | - | MIT |
| **bazi_core** | Dart/Flutter | 2 | 真太阳时、15 种关系类型、刑冲合害 | - | MIT |

### 1.2 推荐方案

**双引擎架构**：

```
┌──────────────────────────────────────┐
│            前端排盘                    │
│  Lunisolar.js（八字四柱 + 节气 + 神煞）│
│  mystilight-8char（十神 + 五行力量）   │
│            ↓                          │
│     前端本地计算，毫秒级响应            │
└──────────────┬───────────────────────┘
               │ 校验请求（可选）
┌──────────────▼───────────────────────┐
│            后端校验                    │
│  bazi-mcp（MCP 协议，独立计算）        │
│  双源对比 → 不一致则报警               │
└──────────────────────────────────────┘
```

**理由**：
1. **前端计算 = 极低延迟 + 离线可用 + 隐私保护**（用户的出生数据不用上传）
2. **后端校验 = 双源验证**，解决"排盘不准"这个行业头号痛点
3. **Lunisolar.js 覆盖基础排盘**，mystilight-8char 补充十神和五行力量的精细计算

### 1.3 排盘核验流程

基于已有 skill 的"排盘核验铁律"，产品化后的流程：

```
用户输入出生日期时辰
        ↓
前端引擎 A（Lunisolar.js）计算四柱
        ↓
前端引擎 B（mystilight-8char）计算四柱
        ↓
    A == B ？
   ┌──┴──┐
   是    否
   ↓     ↓
 继续   后端 bazi-mcp 第三方校验
        ↓
      取多数一致结果
      ↓
    标注"已交叉验证"
```

### 1.4 特殊场景处理

| 场景 | 处理方式 |
|------|---------|
| 节气交接日 | 查询精确节气时刻（小时/分钟级），而非只看日期 |
| 时辰交界 ±30 分钟 | 提示用户"你的出生时间在边界，以下是两种可能的命盘" |
| 早子时 / 晚子时 | 两套都算，让用户根据描述选择更符合的那个 |
| 未提供出生时辰 | 只展示年月日三柱分析，明确标注"时柱未知，部分结论仅供参考" |

## 二、LLM 模型选型

### 2.1 MiMo 系列详细对比

| 模型 | 输入价格 | 输出价格 | Context | 参数 | 特点 |
|------|---------|---------|---------|------|------|
| **MiMo V2 Flash** | $0.10/M | $0.30/M | 256K | 309B/15B 活跃 | 成本最优，MoE 架构 |
| **MiMo V2 Omni** | $0.40/M | $2.00/M | 262K | - | 多模态（文/图/音/视频） |
| **MiMo V2 Pro** | $1.00/M | $3.00/M | 1M input / 16K output | - | 性能最强 |

**对比参照**（同级别西方模型）：
- Claude Opus 4.5 输出价格：$15/M tokens = MiMo Flash 的 **50 倍**
- GPT-4o 输出价格：$15/M tokens = MiMo Flash 的 **50 倍**

### 2.2 成本估算

**场景 1：生成一份完整报告**

| 项目 | Token 估算 | 成本（MiMo Flash） |
|------|-----------|-------------------|
| System Prompt（含 skill 方法论摘要） | ~4K input | $0.0004 |
| 用户命盘 + 典籍引用片段 | ~6K input | $0.0006 |
| 生成报告（3000-5000 字） | ~4K output | $0.0012 |
| **单次总成本** | ~14K tokens | **$0.0022 ≈ ¥0.016** |

**场景 2：对话模式（单轮）**

| 项目 | Token 估算 | 成本（MiMo Flash） |
|------|-----------|-------------------|
| System + 历史上下文 | ~8K input | $0.0008 |
| 用户问题 + 命盘 | ~2K input | $0.0002 |
| AI 回复 | ~1K output | $0.0003 |
| **单轮总成本** | ~11K tokens | **$0.0013 ≈ ¥0.009** |

**场景 3：日活 1 万用户的月成本**

| 假设 | 数据 |
|------|------|
| 日活 | 10,000 |
| 每人每天平均 API 调用次数 | 1.5（含报告和对话） |
| 单次平均成本 | ¥0.012 |
| **日成本** | **¥180** |
| **月成本** | **¥5,400** |

**结论**：使用 MiMo V2 Flash，即使日活万人，月 API 成本也仅约 5000 元。**"cover 掉运营和 token 成本"的目标非常容易实现。**

### 2.3 模型选型建议

| 场景 | 推荐模型 | 理由 |
|------|---------|------|
| 报告生成 | MiMo V2 Flash | 成本极低，输出质量对命理分析足够 |
| 对话模式 | MiMo V2 Flash | 同上 |
| 复杂场景（长上下文对话） | MiMo V2 Omni | 更好的推理能力，成本仍可控 |
| 开发 / 测试阶段 | Claude / GPT-4o | 最强输出质量用于打磨 prompt |

### 2.4 典籍集成方案

已有的 bazi-analysis skill 包含五部经典的完整原文（182 文件），不能全部放入 prompt（太长太贵）。推荐方案：

```
用户输入命盘
      ↓
排盘引擎计算出关键属性
（日主、月令、格局、核心十神、关键合/冲/刑）
      ↓
检索引擎：根据关键属性检索相关典籍章节
（例：日主庚金 + 月令卯木 → 检索穷通宝鉴论庚金、
 子平真诠论正财格、三命通会论庚寅日等）
      ↓
将检索到的典籍片段（约 2-4K tokens）
+ 命盘数据
+ System Prompt（方法论摘要）
一起送入 LLM
      ↓
LLM 生成报告（引用典籍原文 + 白话翻译 + 命盘解读）
```

**关键**：典籍检索用确定性规则（不用 LLM），LLM 只负责"理解 + 表达"。

## 五、关键技术风险

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| **排盘引擎不准** | 全盘分析失效 | 双引擎交叉验证 + 节气精确时刻查询 |
| **LLM 幻觉** | 编造典籍内容 | 典籍检索用规则引擎，LLM 只做表达 |
| **Token 成本失控** | 对话模式可能消耗大量 token | 设置对话轮次上限 / 上下文压缩策略 |
| **微信封杀** | 分享链接被微信识别为"诱导分享" | 避免强制分享解锁内容的设计 |
| **合规审查** | 小程序审核不通过 | MVP 用 H5 绕过，后续再申请小程序 |
| **MiMo 中文命理能力** | 模型对命理术语的理解可能不足 | Prompt 中注入充分的命理上下文 + 典籍片段 |

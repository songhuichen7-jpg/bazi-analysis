# 典型用户行为分析：基于真实使用案例

> 最后更新：2026-04-17
> 数据来源：项目创始人在 AI Agent 算命中的实际使用记录
> 本文档以一位深度用户为样本分析行为模式，提取产品设计启示

---

## 为什么要做这个分析

项目创始人之一本身就是典型的目标用户 -- 用 AI Agent（Cursor/Claude Code）给自己和朋友算命，并且乐在其中。他的使用行为暴露了很多产品设计上需要回答的问题。

---

## 一、行为数据概览

| 维度 | 数据 |
|------|------|
| 独立分析主题 | 8+ 个（感情线、正缘类象、合婚、典籍审阅、多人对比、命盘分析、外貌推断...） |
| Prompt 迭代次数 | 至少 4 个版本（v1 → v3 → v4 + 多个变体） |
| 累计子 Agent 数量 | 单次任务最多 82-84 路，多任务合计数百路 |
| 产出文档 | 30+ 个 markdown 文件，含经典原文、方法论、分析报告、叙事文学 |
| 集中使用时段 | 密集使用数天 |
| 典籍覆盖 | 14 部以上，五部完整原文 |

## 二、关注领域分布

该用户近期的分析高度集中在感情/婚恋方向（~80%），但**这是个案而非普遍规律** -- 他近期在感情方向有较多困扰，这恰好是他此时此刻最 care 的事。

**真正的产品启示不是"感情比事业重要"，而是：**

> **用户在算命时关注什么，取决于他当下人生中最焦虑的事。命理产品的价值在于能精准回应"此时此刻最 care 的事"。**

- 一个正在求职的人 → 事业运、财运
- 一个感情受挫的人 → 感情运、正缘画像
- 一个年初规划的人 → 全年运势概览
- 一个遇到健康问题的人 → 健康运

这意味着产品不应该预设"哪个领域最重要"，而应该：
1. 免费层覆盖所有主要领域（性格 + 事业 + 感情 + 年运），给用户一个全景
2. 付费层让用户自己选择深入哪个方向
3. 对话模式天然解决这个问题 -- 用户问什么就聊什么

---

## 三、使用深度的四个阶段

该用户的行为展现了一个清晰的深度递进路径：

### 阶段 1：基础排盘 → "我是什么命"

- 验证四柱、确认十神、定格局
- 这是每个用户的起点，对应产品的免费层

### 阶段 2：典籍验证 → "古人怎么说我的命"

- 从穷通宝鉴查调候、子平真诠查格局、滴天髓查气势
- 14 部典籍逐条审阅与命主的适用性
- 这一步投入最多精力 -- **说明"有据可查"是核心需求**

### 阶段 3：关系探索 → "对我有意义的人是什么样的"

- 从十神类象推导相关人物特征
- 与现实中的具体人对比验证（合盘、合婚）
- 多 Agent 交叉打分实验
- 这一步产出了最多的"惊喜时刻"

### 阶段 4：叙事化 → "把命盘变成一个故事"

- 半生小说化时间线
- 人物设定表
- 这一步说明**用户最终想要的不是数据，而是关于自己的故事**

**产品启示**：这四个阶段恰好对应产品的四层架构：

```
免费 → 基础排盘 + 命格标签
轻度付费 → 典籍引用的叙事报告
中度付费 → 专题深度解析（按用户当下最关心的方向）
高价值 → 对话模式（持续追问、合盘对比）
```

## 四、"惊喜时刻"分析

从使用记录中，能明确识别出三种"太准了"的体验：

### 类型 1：现实回测吻合

命理推导与用户已知事实吻合时（如："从未喜欢某人、价值观冲突"的命理解释与现实完全一致），用户的信任和兴趣指数级增长。

**这是最强的惊喜来源。**

### 类型 2：结构自洽

多个独立子 Agent 从不同角度得出一致结论时，"交叉验证"本身就是惊喜。

### 类型 3：叙事共鸣

> "不是条件配不上，是一种更深的、说不出来的不踏实感"

当命理结构被翻译成触及内心的表述时，产生的不是"准"而是"被看见"的感觉。

**产品启示**：
- 类型 1 可以设计为"回测验证"功能 -- 用户确认/否认某些推导后，报告进一步优化
- 类型 2 要求排盘和分析过程是可信的（不能用假多 Agent 糊弄）
- 类型 3 是叙事品质的核心指标 -- prompt engineering 的打磨重点

## 五、痛点与不满

从 prompt 的迭代历程中暴露的痛点，直接就是产品需要避免的坑：

### 痛点 1：假多 Agent / "一饰多角"

v4 prompt 明文写："禁止用同一会话扮演 Ag1-Ag8，必须用 Task 足额并行"

**产品启示**：如果产品宣称"多流派交叉验证"，后端就必须真的做到。深度用户能识别伪交叉验证。

### 痛点 2：AI 偷懒不看原始数据

v3/v4 反复强调"必须把原始数据传给子 agent""用 Read 读原始文件"

**产品启示**：典籍引用不能靠 LLM "印象中的古籍"，必须真的检索和读取原文。

### 痛点 3：结论随输入漂移

v3/v4 的"硬隔离墙"：阶段一锁定结论后禁止因后续信息反向修改

**产品启示**：产品化时需要有"先推后验"的流程锁 -- 先生成盲推结果，再与用户已知事实对比，而不是先问用户再"定制"结论。

### 痛点 4：规则口径不统一

不同命理规则体系在同一问题上可能给出不同结论

**产品启示**：排盘和命理规则需要有明确的选择和说明，不能含糊。

### 痛点 5：巴纳姆效应 / 模型保守

用户直接质疑："差异巨大的结论是不是因为巴纳姆效应、模型保守、对齐偏好"

**产品启示**：用户会质疑 AI 是否在"说好听的"。需要有机制证明分析的差异性（比如"你的命盘在 XX 方面与多数人不同"）。

## 六、深度用户画像

基于该用户的行为，补充对 Tier 3（深度咨询型）和 Tier 4（命理爱好者）的画像：

| 维度 | 描述 |
|------|------|
| 年龄 | 大学生，即将毕业 |
| 核心驱动 | 当下最焦虑的人生议题（感情/事业/方向）+ 对"确定性"的渴望 |
| 使用方式 | 不是"问一句运势"，而是**把 AI 当成可审计的研究管线** |
| 对品质的要求 | 证据链 > 感觉；典籍引用 > AI 编造；交叉验证 > 单一输出 |
| 输出偏好 | 硬核可核对（表格 + 引用）与感性可代入（叙事 + 人设）两套并行 |
| 心理需求 | 把模糊焦虑变成可叙述的结构；自我理解与"被看见"；可操作的方向感 |

### 关键发现：Tier 3/4 用户是产品的"灵魂用户"

- 他们不贡献最多的用户量，但贡献最多的：口碑、内容深度要求、付费意愿
- 他们的使用方式会倒逼产品质量 -- 如果连他们都觉得准，大众用户的体验只会更好

## 七、对产品设计的直接启示

| 洞察 | 产品设计指导 |
|------|------------|
| 用户关注什么取决于当下最焦虑的事 | 不预设领域优先级，让用户自选深入方向 |
| 典籍直引 = 信任基石 | 每个结论必须标注来源，支持用户追溯 |
| 叙事 > 数据 | 最终形态是故事，不是表格 |
| 多 Agent 交叉验证被期待 | 如果宣称交叉验证就必须真做，否则不宣称 |
| "先推后验"是核心方法论 | 产品流程：盲推 → 出结果 → 用户确认/否认 → 深化 |
| 回测吻合 = 最强惊喜 | 考虑加入"回测验证"交互 |
| 对巴纳姆效应高度警觉 | 输出必须有差异性指标 |
